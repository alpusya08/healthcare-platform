# MedAI — Pre-Defense Polish Tasks

> **Контекст:** Дипломный MVP. До защиты ~1-2 недели. Все базовые модули уже работают (auth, AI-анализ, бронирование, кабинеты, ML feedback). Нужно довести UX и бизнес-логику до уровня "стыдно не было показать комиссии".
>
> **Стек:** Java 21 + Spring Boot 3.3 (Modulith) / Python 3.11 + FastAPI / React 18 + TypeScript + shadcn/ui + Tailwind / PostgreSQL 16 / MinIO / MLflow.
>
> **Правила работы:**
> - Соблюдать существующую архитектуру (Hexagonal в Java, Clean Architecture в Python, Feature-Sliced в React).
> - DTO ↔ Entity через MapStruct. Никаких JPA-сущностей наружу.
> - Все новые таблицы — через Flyway миграции (V18, V19...).
> - Для каждого нового API — обновить OpenAPI/Swagger.
> - i18n: тексты на русском (это основной язык интерфейса).

### TASK-01: Полноценный функционал расписания и слотов у врача

**Контекст:** Сейчас у всех врачей слоты выглядят одинаково и полностью свободны до 21:00 — это нереалистично и не работает приёмная логика.

**Backend:**
1. Расширить сущность `Schedule` / `TimeSlot`:
   - `appointmentType`: `ONLINE_ONLY`, `OFFLINE_ONLY`, `BOTH` (по умолчанию у врача в профиле, override на уровне слота)
   - `clinicId` (FK → новая таблица `clinics`)
   - `slotDuration`: 15 / 30 / 45 / 60 минут (выбирает врач)
   - `breakBetweenSlots`: 0 / 5 / 10 / 15 минут
2. Новая таблица `clinics`:
   - `id, name, address, city, latitude, longitude, phone, working_hours_json`
3. Связь врач ↔ клиники (many-to-many через `doctor_clinics`):
   - Один врач может принимать в нескольких клиниках в разные дни.
4. Новые эндпоинты для врача:
   - `POST /api/v1/doctor/schedule/template` — шаблон недели (пн-пт 9:00-13:00, 14:00-18:00)
   - `POST /api/v1/doctor/schedule/generate` — сгенерировать слоты на N недель вперёд по шаблону
   - `PATCH /api/v1/doctor/slots/{id}/block` — заблокировать конкретный слот (отпуск, перерыв)
   - `DELETE /api/v1/doctor/slots/{id}` — удалить слот
   - `POST /api/v1/doctor/clinics` — добавить клинику в свой список приёма
5. Seed-миграция (V18): для всех существующих врачей в БД пометить **30-50% слотов как `BOOKED`** случайным образом — чтобы при демо комиссия видела реалистичную загруженность.
   - Создать "технических пациентов" `seed_patient_1..5` для этих фиктивных броней.
   - Часть слотов помечать как `ONLINE_ONLY`, часть — `OFFLINE_ONLY`, часть — `BOTH`.
   - Часовой диапазон: 8:00 — 20:00 (не до 21:00), с обедом 13:00-14:00.

**Frontend — UI врача для управления расписанием:**
- Страница `/doctor/schedule`:
  - Календарь на месяц (компонент типа `react-big-calendar` или `FullCalendar`).
  - Кнопка "Шаблон недели" — модалка где врач задаёт чек-боксы дней недели и временные диапазоны.
  - Toggle "Тип приёма по умолчанию": Онлайн / Офлайн / Оба.
  - Селектор длительности слота: 15/30/45/60 мин.
  - Раздел "Мои клиники" — список + кнопка "Добавить клинику" (открывает форму или поиск по существующим).
  - На клик по слоту в календаре — попап с возможностью заблокировать / разблокировать / изменить тип.

**Acceptance:**
- Врач может за 1 минуту настроить своё расписание на 4 недели вперёд.
- В списке врачей у каждого видна реалистичная картина: есть занятые слоты, есть свободные.
- Врач может указать "принимаю только онлайн" — и в его карточке пациент видит только онлайн-слоты.

### TASK-04: Booking flow прямо со страницы врача

**Контекст:** Сейчас на странице врача слоты не кликабельны для прямого бронирования.

**Желаемый flow:**
1. Пациент на странице врача видит сетку ближайших слотов (4 ближайших рабочих дня, по 6-8 слотов на день).
2. Клик на свободный слот → слот подсвечивается как выбранный.
3. Появляется sticky-кнопка внизу "Записаться на 15 мая, 14:30".
4. Клик на кнопку → модалка / шаг 2:
   - Радио: Онлайн / Офлайн (только варианты, которые врач поддерживает в этом слоте).
   - Textarea: "Причина визита / жалоба" (если пациент пришёл с AI-отчёта — предзаполнено).
   - Чекбокс "Прикрепить мой последний AI-отчёт к записи".
5. Кнопка "Подтвердить запись" → POST `/api/v1/appointments`.
6. Toast "Запись создана" + редирект в кабинет → список приёмов.

**Технические детали:**
- Слот блокируется на сервере на 5 минут на время оформления (`status = PENDING_RESERVATION`).
- Если пациент не подтвердил за 5 минут — слот возвращается в `AVAILABLE`.
- Для онлайн — после `CONFIRMED` генерируется Jitsi-ссылка вида `https://meet.jit.si/medai-{appointmentId}`.

**Frontend изменения:**
- На странице `/doctors/{id}` — переработать секцию "Расписание" в кликабельную сетку.
- Сохранить текущий код кнопки "Записаться" как fallback (если слот не выбран — кнопка дизейблена с подсказкой "Выберите время").

### TASK-05: AI Analysis — распознавание прикреплённых файлов и адаптация вопросов

**Контекст:** Сейчас файл можно прикрепить, но LLM на него не реагирует. Нужно сделать так чтобы прикрепление ЭКГ/УЗИ/заключения меняло ход анкетирования.

**AI Service (Python):**
1. Расширить `file_processor`:
   - PDF → `pypdf` извлекает текст → передаётся в Claude как контекст.
   - JPG/PNG (фото заключения, ЭКГ) → Claude Vision API (или GPT-4 Vision как fallback) → описание изображения и извлечение текста.
   - DICOM → пока только preview + название файла как контекст (полный анализ вне скоупа MVP).
2. Новый шаг в pipeline `feature_extractor`:
   - После обработки файла — отправить его содержимое в LLM с промптом:
   ```
   Извлеки из этого медицинского документа структурированные признаки:
   - Тип документа (ЭКГ / УЗИ / заключение / анализ крови / другое)
   - Дата документа
   - Ключевые показатели (если есть числа — холестерин, ЧСС, давление и т.д.)
   - Уже поставленный диагноз (если указан)
   - Рекомендации врача (если указаны)
   Верни строго JSON.
   ```
3. Эти извлечённые признаки **сразу подставляются в `MedicalFeatures`** сессии — и `generate_next_question` уже учитывает их и не спрашивает то, что уже есть в файле.
4. В UI на этапе генерации вопроса — показать badge "На основе вашего заключения от 12.03.2026 я уже знаю что..." чтобы пациент видел что система прочитала файл.

**Backend (Java):**
- Эндпоинт `POST /ai/sessions/{id}/files` — принимает файл, передаёт в AI-service, сохраняет ссылку в MinIO.
- AI-service возвращает извлечённые признаки → Java сохраняет их в `AnalysisSession.extractedFeatures` (jsonb).

**Frontend:**
- После загрузки файла — спиннер "Анализирую документ..." (3-7 секунд).
- После успеха — карточка с превью: "📄 Заключение кардиолога, 12.03.2026 — извлечены данные: холестерин 280, ЧСС покоя 78".
- Следующий вопрос — с пометкой "Учитываю данные из заключения".

**Acceptance:**
- При загрузке заключения с указанным холестерином — LLM не задаёт вопрос про холестерин снова.
- Пациент видит что система прочитала файл (это must-have для wow-эффекта на демо).
